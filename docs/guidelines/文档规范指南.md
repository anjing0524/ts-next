# 文档规范指南

> **文档版本**: v3.0.0  
> **最后更新**: 2025-07-22  
> **维护团队**: 开发团队

## 核心原则

- **简洁性**: 避免冗余，直接表达核心信息
- **一致性**: 统一格式、命名和结构
- **实用性**: 专注于实际开发需要的信息
- **时效性**: 保持文档与代码同步更新

## 1. 命名规范

### 1.1 文件命名

- 使用中文名称，描述清晰
- 文件名格式：`功能模块-文档类型.md`
- 示例：`系统架构-设计文档.md`、`用户管理-API文档.md`

### 1.2 目录结构

```
docs/
├── requirements/     # 需求文档
├── design/          # 设计文档
├── guidelines/      # 规范文档
├── ops/            # 运维文档
├── tests/          # 测试文档
└── generated/      # 自动生成文档
```

## 2. 文档模板

### 2.1 通用模板

```markdown
# [文档标题]

> **文档版本**: v1.0.0  
> **最后更新**: 2025-07-22  
> **维护团队**: [团队名称]

## 概述

[文档目的和主要内容]

## 详细内容

[具体章节内容]
```

### 2.2 设计文档模板

```markdown
# [系统/模块]设计文档

> **文档版本**: v1.0.0  
> **最后更新**: 2025-07-22

## 1. 设计概述
## 2. 系统架构
## 3. 详细设计
## 4. 技术实现
```

### 2.3 API文档模板

```markdown
# [模块]API接口文档

> **文档版本**: v1.0.0  
> **最后更新**: 2025-07-22

## 接口列表

### [接口名称]

**请求方式**: GET/POST/PUT/DELETE  
**请求路径**: /api/v1/example  

#### 请求参数

| 参数名 | 类型   | 必填 | 描述     |
| ------ | ------ | ---- | -------- |
| param1 | string | 是   | 参数描述 |

#### 响应示例

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```
```

## 3. 基本规范

- 使用中文编写，技术术语可标注英文
- 使用Markdown格式
- 中英文之间添加空格
- 代码块使用三个反引号包围
- 图片存储在 `/docs/images/` 目录
